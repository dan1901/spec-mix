# 프로젝트 헌장: [프로젝트 이름]

## 핵심 원칙

### 1. 코드 품질

- 깨끗하고, 읽기 쉬우며, 유지보수 가능한 코드 작성

- 확립된 코딩 규칙 및 스타일 가이드 준수

- 영리함보다 명확성 우선

### 2. 클린 아키텍처

- **의존성 규칙 준수**: 외부 계층에서 내부 계층으로만 의존 (Framework → Interface Adapter → Use Case → Entity)

- **계층 분리**:
  - Entity: 핵심 비즈니스 규칙 (외부 의존성 없음)
  - Use Case: 애플리케이션 비즈니스 로직
  - Interface Adapter: 컨트롤러, 프레젠터, 게이트웨이
  - Framework & Drivers: 웹, DB, 외부 라이브러리

- **관심사 분리**: 각 모듈/클래스는 단일 책임 원칙(SRP) 준수

- **테스트 가능성**: 비즈니스 로직은 프레임워크/DB 없이 단위 테스트 가능해야 함

- **프레임워크 독립성**: 핵심 비즈니스 로직은 특정 프레임워크에 종속되지 않음

### 3. 테스트 기준

- 모든 새 기능에는 테스트 포함 필수

- 높은 테스트 커버리지 유지 (>80% 목표)

- 엣지 케이스 및 오류 조건 테스트

- 예상 동작을 문서화하는 테스트 작성

### 4. 사용자 경험

- 최종 사용자를 염두에 두고 설계

- 애플리케이션 전체의 일관성 보장

- 명확한 오류 메시지 및 피드백 제공

- 직관적이고 접근 가능한 인터페이스 만들기

### 5. 성능 요구사항

- 일반적인 사용 사례에 최적화

- 최적화 전 프로파일링

- 성능 중요 섹션 문서화

- 성능과 유지보수성 균형

### 6. 문서화

- 공개 API 및 인터페이스 문서화

- 문서에 예제 포함

- README 최신 상태 유지

- 아키텍처 결정사항 문서화

### 7. 보안

- 모든 사용자 입력 검증

- 보안 모범 사례 준수

- 의존성 최신 상태 유지

- 취약점에 대한 코드 검토

### 8. 협업

- 설명적인 커밋 메시지 작성

- 코드를 신중하고 건설적으로 검토

- 결정 및 트레이드오프 소통

- 팀과 지식 공유

## 개발 워크플로우

1. **명세(Specification)**: 무엇을 만들지 정의

1. **계획(Planning)**: 기술 구현 계획 작성

1. **구현(Implementation)**: 테스트와 함께 구축

1. **검토(Review)**: 코드 및 테스트 피어 리뷰

1. **통합(Integration)**: main 브랜치에 병합

1. **배포(Deployment)**: 사용자에게 릴리스
