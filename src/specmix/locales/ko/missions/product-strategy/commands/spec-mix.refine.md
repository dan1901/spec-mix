## 사용자 입력

```text
{ARGS}
```

위 입력이 비어있지 않다면 반드시 고려해야 합니다.

## 개요

이 명령어는 이해관계자 피드백, 분석 결과, 또는 새로운 정보를 반영하여 6-Pager 문서를 개선합니다.

---

## 작업 절차

### 1단계: 현재 문서 확인

```bash
# 6-Pager 문서 읽기
cat specs/strategy/6pager.md

# 분석 문서 확인 (있는 경우)
ls -la specs/strategy/*.md
```

### 2단계: 피드백 유형 확인

사용자 입력을 분석하여 피드백 유형을 파악합니다:

| 유형 | 예시 | 처리 방식 |
|------|------|-----------|
| **섹션 수정** | "시장 크기 업데이트해줘" | 해당 섹션만 수정 |
| **데이터 업데이트** | "TAM을 $10B로 변경" | 수치 업데이트 |
| **피드백 반영** | "검토자가 경쟁사 분석 보완 요청" | 해당 내용 보강 |
| **구조 변경** | "페르소나 하나 더 추가" | 구조 확장 |
| **전체 개선** | (빈 입력) | 전체 검토 후 개선점 제안 |

### 3단계: 피드백에 따른 처리

#### A. 특정 섹션 수정 요청

1. 해당 섹션 확인
2. 요청사항 반영
3. 변경 내용 요약 제시

```markdown
## 변경 완료

### 수정된 섹션: [섹션명]

**변경 전**:
[이전 내용 요약]

**변경 후**:
[새로운 내용 요약]

### 변경 사유
[왜 이렇게 변경했는지]
```

#### B. 분석 결과 반영 요청

분석 문서의 내용을 6-Pager에 통합합니다:

```bash
# 분석 결과 읽기
cat specs/strategy/market-research.md 2>/dev/null
cat specs/strategy/competitor-analysis.md 2>/dev/null
cat specs/strategy/customer-analysis.md 2>/dev/null
```

통합 후:

```markdown
## 분석 결과 반영 완료

다음 분석 결과가 6-Pager에 반영되었습니다:

### 시장 분석 → 섹션 4 (시장 크기)
- TAM/SAM/SOM 수치 업데이트
- 시장 성장률 데이터 추가

### 경쟁사 분석 → 섹션 6 (경쟁사 비교)
- 경쟁사 매트릭스 보완
- 차별화 포인트 명확화

### 고객 분석 → 섹션 5 (고객 이해)
- 페르소나 검증 내용 반영
- Pain Point 우선순위 조정
```

#### C. 전체 개선 (빈 입력)

문서 전체를 검토하고 개선점을 제안합니다:

```markdown
## 6-Pager 품질 검토

### 현재 상태 평가

| 섹션 | 완성도 | 주요 이슈 |
|------|--------|-----------|
| 1. 한 줄 요약 | ⭐⭐⭐⭐⭐ | - |
| 2. 배경과 목적 | ⭐⭐⭐⭐ | Why Now 보강 필요 |
| 3. 목표 & KPI | ⭐⭐⭐ | KPI 구체화 필요 |
| ... | | |

### 개선 권장 사항

#### 우선순위 높음
1. **[섹션명]**: [개선 내용]
2. **[섹션명]**: [개선 내용]

#### 우선순위 중간
1. **[섹션명]**: [개선 내용]

#### 우선순위 낮음
1. **[섹션명]**: [개선 내용]

### 추가로 필요한 정보

| 항목 | 필요한 이유 | 확보 방법 |
|------|------------|-----------|
| | | |

개선을 진행하시겠습니까? (Y/N)
```

### 4단계: 변경 이력 기록

문서 하단의 "변경 이력" 섹션을 업데이트합니다:

```markdown
### C. 변경 이력

| 버전 | 날짜 | 변경 내용 | 작성자 |
|------|------|-----------|--------|
| v1.0 | YYYY-MM-DD | 최초 작성 | [이름] |
| v1.1 | YYYY-MM-DD | [변경 사항 요약] | AI Assistant |
```

### 5단계: 결과 제시

```markdown
## 개선 완료

### 변경 요약

| 섹션 | 변경 유형 | 설명 |
|------|-----------|------|
| | 수정 | |
| | 추가 | |
| | 삭제 | |

### 다음 단계

- [ ] 변경 내용 검토
- [ ] 이해관계자 피드백 수집
- [ ] `/spec-mix.review` 로 공식 리뷰 진행
```

## 개선 가이드라인

### 좋은 개선의 원칙

1. **일관성 유지**: 문서 전체의 톤과 스타일 유지
2. **변경 추적**: 무엇이 왜 변경되었는지 명확히 기록
3. **근거 명시**: 데이터 변경 시 출처 업데이트
4. **영향 범위 확인**: 한 섹션 변경이 다른 섹션에 미치는 영향 검토

### 주의사항

- 승인된 내용은 함부로 변경하지 않음
- 숫자 변경 시 관련 계산도 함께 업데이트
- 가정(Assumption)이 검증되면 표시 제거
- 새로운 가정 추가 시 명확히 표시

### 버전 관리

- Minor 변경: v1.0 → v1.1
- Major 변경: v1.1 → v2.0
- 리뷰 후 승인: Draft → v1.0

## 자주 사용하는 개선 패턴

### 데이터 업데이트

```
입력: "TAM을 $50B에서 $75B로 변경, 출처는 Gartner 2024"

처리:
1. 섹션 4의 TAM 값 업데이트
2. 산출 근거에 출처 추가
3. SAM/SOM 비율 재검토
4. 변경 이력에 기록
```

### 피드백 반영

```
입력: "CFO가 Unit Economics 섹션에 더 상세한 CAC breakdown 요청"

처리:
1. 섹션 9.3의 CAC 항목 확장
2. 채널별 CAC breakdown 추가
3. 관련 가정 명시
4. 변경 이력에 기록
```

### 경쟁사 추가

```
입력: "경쟁사 D 추가해줘"

처리:
1. 사용자에게 경쟁사 D 정보 요청
2. 섹션 6의 경쟁사 매트릭스에 추가
3. 경쟁 우위 분석 업데이트
4. 변경 이력에 기록
```
